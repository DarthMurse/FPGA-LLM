# BitNet FPGA RTL Interface Spec

## 1. Goal

This document defines the first-pass interfaces between host software and FPGA RTL modules.

The goals are:

- make module ownership explicit
- make simulation and integration easier
- keep the first implementation simple
- freeze the control/data contracts before RTL coding

This is a first-pass interface spec. Numeric precision and exact buffer depths may still change later.

## 2. Scope

This document covers:

- host-to-FPGA Ethernet protocol
- boot-time model initialization control
- FPGA top-level module boundaries
- control handshakes between major blocks
- data movement conventions

This document does not yet freeze:

- exact fixed-point formats
- exact ternary weight packing
- exact DDR burst lengths
- exact BRAM depths

Those belong in `docs/memory_layout.md`.

## 3. Interface Design Rules

All major RTL modules should follow these rules:

- synchronous to a single local `clk`
- active-low reset `rst_n`
- explicit `start/busy/done` control for coarse-grain operations
- `valid/ready` style handshake for streams
- side effects must be visible at module boundaries
- module outputs must remain stable while `valid=1` and `ready=0`

Naming conventions:

- inputs use `_i`
- outputs use `_o`
- handshake signals use `_valid`, `_ready`
- addresses use `_addr`
- lengths use `_len`
- byte counts use `_bytes`
- metadata IDs use `_id`

## 4. System-Level Dataflow

The runtime loop is:

1. Host tokenizes text.
2. Host converts token IDs into embedding vectors.
3. Host sends embedding data to FPGA over Ethernet.
4. FPGA runs transformer layers and maintains KV cache.
5. FPGA returns final hidden state to host.
6. Host runs final RMSNorm, LM head, and sampling.
7. Host sends the next embedding for decode.

This means the FPGA-side external payloads are:

- control commands
- embedding vectors in
- final hidden-state vectors out
- status and debug messages

Weight loading is not part of the Ethernet runtime path.
Weights, scales, and metadata are initialized from SD card or QSPI into DDR before inference begins.

## 5. Host Ethernet Protocol

## 5.1 Transport choice

First-pass recommendation:

- use Gigabit Ethernet
- use a simple packet format
- prefer reliability and debuggability over maximum throughput

First-pass software approach:

- host runtime sends one request at a time
- FPGA returns one response at a time
- no out-of-order completion

## 5.2 Packet framing

Each application packet should contain:

- `msg_type`
- `flags`
- `sequence_id`
- `payload_bytes`
- `session_id`
- payload

Suggested fixed header fields:

```text
31:24  msg_type
23:16  flags
15:0   header_bytes
31:0   sequence_id
31:0   session_id
31:0   payload_bytes
31:0   reserved
```

Notes:

- `sequence_id` is generated by the host and echoed by FPGA responses.
- `session_id` lets the host reset or restart inference cleanly.
- `payload_bytes` excludes link-layer and transport-layer overhead.

## 5.3 Message types

Recommended first-pass message types:

- `0x01` `MSG_RESET`
- `0x02` `MSG_LOAD_MODEL_INFO`
- `0x03` `MSG_PREFILL_EMBED_TILE`
- `0x04` `MSG_PREFILL_RUN`
- `0x05` `MSG_DECODE_EMBED`
- `0x06` `MSG_DECODE_RUN`
- `0x07` `MSG_READ_STATUS`
- `0x08` `MSG_READ_DEBUG`
- `0x81` `MSG_ACK`
- `0x82` `MSG_ERROR`
- `0x83` `MSG_PREFILL_RESULT`
- `0x84` `MSG_DECODE_RESULT`
- `0x85` `MSG_STATUS`
- `0x86` `MSG_DEBUG`

## 5.4 Payload definitions

### `MSG_RESET`

Purpose:

- resets runtime state on FPGA
- invalidates current inference session

Payload:

- optional reset flags

Response:

- `MSG_ACK`

### `MSG_LOAD_MODEL_INFO`

Purpose:

- loads runtime metadata that is small and changes per run

First-pass fields:

- `hidden_size`
- `num_layers`
- `num_heads`
- `num_kv_heads`
- `max_context_used`
- `embed_elem_bytes`
- `hidden_elem_bytes`
- `activation_format_id`
- `activation_scale_format_id`

Response:

- `MSG_ACK` or `MSG_ERROR`

### `MSG_PREFILL_EMBED_TILE`

Purpose:

- sends one tile of prompt embeddings from host to FPGA

Header metadata:

- `tile_id`
- `token_start`
- `token_count`
- `embed_dim`
- `elem_bytes`

Payload:

- packed quantized embedding vectors for `token_count` prompt positions
- one `fp16` activation scale per token vector

Response:

- `MSG_ACK`

### `MSG_PREFILL_RUN`

Purpose:

- instructs FPGA to run prefill using the previously loaded embedding tile

Header metadata:

- `tile_id`
- `is_last_tile`
- `prompt_length`

Response:

- `MSG_PREFILL_RESULT` for the last tile
- `MSG_ACK` for non-final tiles if no return vector is needed

### `MSG_DECODE_EMBED`

Purpose:

- sends one decode-step embedding vector

Header metadata:

- `position_idx`
- `embed_dim`
- `elem_bytes`

Payload:

- one quantized embedding vector
- one `fp16` activation scale

Response:

- `MSG_ACK`

### `MSG_DECODE_RUN`

Purpose:

- instructs FPGA to run one decode step

Header metadata:

- `position_idx`
- `context_used`

Response:

- `MSG_DECODE_RESULT`

### `MSG_PREFILL_RESULT`

Purpose:

- returns the final hidden state for the last prompt token

Header metadata:

- `token_idx`
- `hidden_dim`
- `elem_bytes`

Payload:

- one quantized hidden-state vector
- one `fp16` activation scale

### `MSG_DECODE_RESULT`

Purpose:

- returns the final hidden state for one decode step

Header metadata:

- `position_idx`
- `hidden_dim`
- `elem_bytes`

Payload:

- one quantized hidden-state vector
- one `fp16` activation scale

### `MSG_READ_STATUS` and `MSG_STATUS`

Purpose:

- poll FPGA progress and error state

Recommended status fields:

- `session_id`
- `busy`
- `mode`
- `current_layer`
- `current_position`
- `error_code`

### `MSG_READ_DEBUG` and `MSG_DEBUG`

Purpose:

- fetch counters or selected internal state for debugging

Recommended debug fields:

- packet counters
- DDR transaction counters
- cycle counters
- last error code

## 5.5 First-pass protocol constraints

To keep bring-up simple:

- host must not issue a new run command while FPGA is busy
- FPGA processes one command at a time
- host retries on timeout
- FPGA returns one explicit completion packet per run command

## 6. Top-Level RTL Module Interfaces

## 6.1 `top_system`

Purpose:

- board-level integration only

Suggested top-level ports:

```verilog
module top_system (
    input  wire        sys_clk_i,
    input  wire        sys_rst_n_i,
    input  wire        kv_clear_btn_i,

    input  wire        eth_rx_clk_i,
    input  wire [7:0]  eth_rxd_i,
    input  wire        eth_rx_dv_i,
    output wire        eth_tx_clk_o,
    output wire [7:0]  eth_txd_o,
    output wire        eth_tx_en_o,

    output wire [13:0] ddr_addr_o,
    output wire [2:0]  ddr_ba_o,
    inout  wire [15:0] ddr_dq_io,
    inout  wire [1:0]  ddr_dqs_p_io,
    inout  wire [1:0]  ddr_dqs_n_io
);
```

Notes:

- exact Ethernet PHY interface depends on board wiring and chosen MAC/IP
- exact DDR3 port list depends on the generated memory controller wrapper

## 6.2 `boot_if`

Purpose:

- initializes DDR contents from SD card or QSPI before runtime inference begins

Suggested coarse-grain ports:

```verilog
module boot_if (
    input  wire        clk_i,
    input  wire        rst_n_i,
    input  wire        boot_start_i,
    output wire        boot_busy_o,
    output wire        boot_done_o,
    output wire        boot_error_o,

    output wire        ddr_wr_cmd_valid_o,
    input  wire        ddr_wr_cmd_ready_i,
    output wire [31:0] ddr_wr_addr_o,
    output wire [15:0] ddr_wr_len_o,
    output wire        ddr_wr_data_valid_o,
    input  wire        ddr_wr_data_ready_i,
    output wire [63:0] ddr_wr_data_o,
    output wire        ddr_wr_data_last_o
);
```

## 6.3 `host_if`

Purpose:

- terminates the Ethernet-side application protocol
- buffers incoming embeddings
- returns hidden-state outputs
- exposes a simple command/data interface to the controllers

Major sub-interfaces:

- Ethernet ingress and egress
- command output toward controllers
- embedding stream output toward controllers
- result stream input from controllers
- status/debug register access

Suggested coarse-grain control ports:

```verilog
module host_if (
    input  wire         clk_i,
    input  wire         rst_n_i,

    output wire         cmd_valid_o,
    input  wire         cmd_ready_i,
    output wire [7:0]   cmd_opcode_o,
    output wire [31:0]  cmd_session_id_o,
    output wire [31:0]  cmd_sequence_id_o,
    output wire [31:0]  cmd_arg0_o,
    output wire [31:0]  cmd_arg1_o,
    output wire [31:0]  cmd_arg2_o,

    output wire         data_out_valid_o,
    input  wire         data_out_ready_i,
    output wire [63:0]  data_out_o,
    output wire         data_out_last_o,

    input  wire         data_in_valid_i,
    output wire         data_in_ready_o,
    input  wire [63:0]  data_in_i,
    input  wire         data_in_last_i,

    input  wire         rsp_valid_i,
    output wire         rsp_ready_o,
    input  wire [7:0]   rsp_opcode_i,
    input  wire [31:0]  rsp_session_id_i,
    input  wire [31:0]  rsp_sequence_id_i,
    input  wire [31:0]  rsp_status_i
);
```

Notes:

- `data_out_*` is host-to-FPGA payload data
- `data_in_*` is FPGA-to-host payload data
- `cmd_opcode_o` represents decoded application commands
- 64-bit streaming width is a reasonable first-pass internal payload width
- model weights are not streamed through this interface during inference

## 6.4 `prefill_controller`

- owns prefill command sequencing

```verilog
    input  wire [31:0]  cmd_token_count_i,
    input  wire         cmd_is_last_tile_i,

    input  wire         embed_valid_i,
    output wire         embed_ready_o,
    input  wire [63:0]  embed_data_i,
    input  wire         embed_last_i,

    output wire         path_start_o,
    input  wire         path_busy_i,
    input  wire         path_done_i,

    output wire         result_valid_o,
    input  wire         result_ready_i,
    output wire [63:0]  result_data_o,
    output wire         result_last_o,

    output wire         rsp_valid_o,
    input  wire         rsp_ready_i,
    output wire [7:0]   rsp_opcode_o,
    output wire [31:0]  rsp_session_id_o,
    output wire [31:0]  rsp_sequence_id_o,
    output wire [31:0]  rsp_status_o

```

Behavior:

- accepts one embedding tile
- launches prefill processing
- for the last tile, streams the last-token hidden state back to `host_if`

## 6.11 `onchip_buffer_bank`

Purpose:

- centralizes local activation and tile storage

Interface intent:

- explicit write and read ports
- separate logical banks for:
  - current hidden state
  - next hidden state
  - prompt tile storage
  - Q/K/V scratch
  - accumulation scratch

First-pass rule:

- submodules should access local buffers through named bank interfaces rather than through ad hoc private RAMs

## 8. Command Routing Rules

To avoid ambiguity:

- `host_if` decodes Ethernet packets into internal commands
- only one of `prefill_controller` or `decode_controller` may accept a command at a time
- `prefill_controller` accepts only prefill-related opcodes
- `decode_controller` accepts only decode-related opcodes
- results must be tagged with the originating `session_id` and `sequence_id`

## 9. Backpressure Rules

These rules apply across all stream interfaces:

- producer may assert `valid` before consumer asserts `ready`
- transfer occurs only when `valid && ready`
- producer must hold data and `last` stable until transfer occurs
- consumer may deassert `ready` at any time
- no data may be dropped silently

## 10. Error Handling

Minimum first-pass error codes:

- `0x0000` success
- `0x0001` unknown opcode
- `0x0002` bad session
- `0x0003` bad payload length
- `0x0004` busy conflict
- `0x0005` unsupported mode
- `0x0006` DDR transaction error
- `0x0007` internal timeout
- `0x0008` invalid dimension

Rule:

- every failed command should produce an explicit `MSG_ERROR`
